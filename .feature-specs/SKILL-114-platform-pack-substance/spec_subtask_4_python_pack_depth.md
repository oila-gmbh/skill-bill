# SKILL-114 Subtask 4 - Python Pack Depth

## Scope

Elevate Python from a useful mid-depth pack to a complete expert review surface
across all ten areas. Keep framework examples conditional and cover Python as a
language/runtime, packaging ecosystem, service/data platform, and UI-capable
stack.

Required coverage includes mutation/aliasing and dataclass/model semantics,
typing/runtime-validation boundaries, exceptions/resources, asyncio task and
cancellation ownership, threads/processes/GIL-sensitive work, imports and
package boundaries, packaging/lock/build backends, Django/FastAPI/Flask and
worker frameworks when detected, SQLAlchemy/Django ORM and migrations,
serialization/template/path/upload/subprocess/network security, pytest and
property/integration evidence, retries/queues/shutdown/telemetry, data/notebook
surfaces, server-rendered/admin/desktop UI, and accessibility.

Quality checking must cover discovered project tools, environment/lock mode,
format/lint/type checking, tests and async tests, packaging/build validation,
dependency/security checks, migration/framework checks when detected, and
targeted-to-full escalation.

## Acceptance Criteria

1. All ten Python specialists meet the depth and concrete-evidence gates.
2. Platform correctness, architecture, performance, and reliability cover
   Python mutation/model semantics, import/package ownership, asyncio
   cancellation/tasks, resource lifetime, concurrency/executor choices,
   queues/workers/retries/shutdown and observable failure paths.
3. API, persistence, security, and testing cover runtime validation, framework
   request contracts, ORM/session/transaction/migration semantics, unsafe
   parsing/templates/uploads/paths/process/network access, and high-value
   pytest/integration/property/async evidence.
4. UI and UX/accessibility address Python-owned admin, template, notebook,
   dashboard and desktop surfaces with rendering, forms, focus/keyboard,
   semantics, feedback and localization concerns.
5. The quality checker handles the detected package/environment manager and
   repository commands, then format, lint, typing, tests, build/package,
   dependency/security and framework-specific validation in safe order.
6. Python passes the 35% pack and 65% corresponding-rubric duplication gates.
7. Metadata, agents, tests, and history are synchronized with the completed
   platform coverage.
8. Python pack tests, `skill-bill validate`, and relevant Gradle checks pass.

## Non-Goals

- No embedded documentation for every Python data or web framework.
- No assumption that typing alone validates runtime inputs.
- No generated provider output committed as source.
